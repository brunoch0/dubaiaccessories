# 기획 세밀화 프롬프트 (시스템 투입용)

파트너 답변(2026-07-23) 반영 완료 — `[파트너 답변]` 채워진 상태라 바로 실행 가능.
아래 프롬프트를 복사해서 기획 세션에 넣고 `Phase {N}`만 지정하면 됨.

## 마스터 프롬프트

```
너는 리테일 재고관리 + 이커머스 시스템 기획 전문가다.
두바이에서 오프라인 액세서리 멀티샵 2곳을 운영하는 파트너를 위한
시스템을 단계별로 기획한다.

[사업 컨텍스트]
- 중국 생산 제품 도매 수입 → 오프라인 2개 매장 판매
  · MCC(Mirdif City Centre) ~1,500 SKU / MOE(Mall of the Emirates) ~1,000 SKU (일부 중복)
- 브랜드/인스타: @whisperofspring (성장 중), 자체 웹사이트 없음
- POS 등록 고객 4,000명 (현장 가입) — 온라인 런칭 시 마케팅 자산
- 핵심 문제: ① 판매 데이터를 매일 수기로 엑셀 작성(POS 무료버전이라
  판매 CSV가 유료) → 업무 과다·오류 ② 온라인 판매 시스템 부재
- 장기 방향: 큐레이션 멀티샵 브랜딩 → 매장 확장 → 자체 브랜드(중장기) → 데이터 인사이트
- 시스템별 수익 연결: 재고관리=🔴방어(수기 업무 제거+확장 전제),
  쇼핑몰=🟢확장(신규 매출), 큐레이션=🔵자산(검색유입·브랜드 신뢰)

[구축 로드맵 — 이 순서는 확정]
- Phase 0: 상품 마스터 데이터 정제 + Loyverse API 연동 확인
- Phase 1: 내부 재고관리 시스템 (매장별 재고, 입고/판매/이동/반품/폐기, Loyverse 자동 싱크)
- Phase 2: B2C 쇼핑몰 (카드 결제 우선, 배송, 실시간 재고 차감) ← 파트너 확정
- Phase 3: B2B 파트너 채널 (도매가, 주문/견적)
- Phase 4: 큐레이션/브랜딩 페이지 + 판매 데이터 대시보드

[기술 제약]
- 스택: Lovable(React) + Supabase(PostgreSQL). 노코드/로우코드 우선
- 매장 직원용 화면은 모바일 우선, 영어 UI (직원 다국적)
- POS: Loyverse 무료버전 사용 중. Integrations 애드온($9/월/매장) 활성화 후
  Loyverse API로 판매 데이터 자동 수집이 기본안. 폴백: 재고 CSV(무료) 수동 업로드
- 운영자는 개발자가 아님 → 관리 화면은 단순하게
- 권한 2계층 (확정 요구사항): Owner/Admin(오너+관리자)만 원가·마진·매출·고객 데이터
  접근 가능. Staff(매장 직원)는 재고 조회/입출고 입력·품절보고만 — 민감 데이터는
  화면 미노출 + Supabase RLS로 DB 레벨에서 차단. 모든 산출물(스키마·화면·RLS)에
  이 2계층을 반영하라

[파트너 답변 — 2026-07-23 확정]
① POS: Loyverse (무료버전)
② 데이터 추출: 재고 CSV 무료 / 판매 CSV 유료 → 현재 판매는 수기 엑셀.
   샘플 파일(수기 판매 엑셀 + 재고 CSV) 공유 예정
③ API: 파트너는 미인지 → 우리가 Integrations 애드온으로 처리
④ 재고 파일: 수기 엑셀 + Loyverse 재고 CSV (샘플 수신 대기)
⑤ SKU/바코드: 자체 규칙 존재 — 연도4+입고일4+카테고리2(01귀걸이/02목걸이/
   03팔찌/04반지/05기타)+순번2(1~10) 입력 → 체크숫자 1자리 자동생성 = 13자리.
   바코드에서 SKU 추출. ⚠️순번 2자리·카테고리 5종 제한은 확장 시 병목 — 보완안 필요
⑥ SKU 규모: MCC ~1,500 / MOE ~1,000 (중복 존재 → 통합 마스터 필요)
⑦ 우선순위: B2C 먼저 (확정)
⑧ 결제: 1순위 카드, 2순위 계좌이체, 3순위 COD
⑨ 자체 브랜드: 중장기
⑩ 인스타: @whisperofspring
⑪ 데이터 공유 범위: 대면 미팅에서 확정 (미정으로 처리)

[요청 — Phase {N}에 대해 아래 산출물을 만들어라]
1. 기능 정의서: 기능별 목적 / 사용자(사장·직원·고객·B2B파트너) / 우선순위(Must·Should·Later)
2. DB 스키마: 테이블, 컬럼, 관계, 핵심 제약조건 (Supabase 기준, RLS 정책 포함)
   — 기존 13자리 바코드/SKU 체계를 수용하되 내부 PK는 분리할 것
3. 화면 목록: 화면별 핵심 요소와 사용자 플로우 (모바일/데스크톱 구분)
4. 재고 이동 규칙: 각 이벤트(입고/판매/이동/반품/폐기/온라인주문)가 재고 수량에
   미치는 영향을 상태 전이표로 — Loyverse 판매 싱크와 온라인 주문 동시 발생 케이스 포함
5. 결정 필요 리스트: 파트너가 정해야 할 정책들 (예: 마이너스 재고 허용 여부,
   온라인 주문 재고 예약 시간, 중복 상품 통합 기준, SKU 보완 규칙 적용 시점)
6. Lovable 초기 빌드 프롬프트: 위 내용을 반영한 첫 프롬프트 초안

산출물은 표 중심으로, 구현 담당자가 그대로 작업 가능한 수준으로 작성하라.
모호한 부분은 가정하지 말고 "결정 필요 리스트"로 분리하라.
```

## 사용법

| 시점 | 실행 |
|------|------|
| 지금 | `Phase 1`로 실행 → 재고 시스템 상세 기획 (답변 다 채워져 있음) |
| 샘플 파일 수신 후 | 샘플 구조를 프롬프트에 추가하고 Phase 0~1 기획 보정 |
| Phase 1 가동 후 | 같은 프롬프트 `Phase 2`(B2C), `Phase 3`(B2B) 순차 실행 |

## Phase 0 전용 짧은 프롬프트 (샘플 파일 도착 전 실행 가능)

```
위 컨텍스트 기준, Phase 0만 진행한다.
1. 상품 마스터 엑셀 템플릿 설계 (컬럼 정의 + 입력 규칙 + 예시 3행)
   — 기존 13자리 바코드 체계 수용 + 두 매장 중복 상품 통합 방법 포함
2. SKU 채번 규칙 보완안 2~3개 비교 (순번 확장·카테고리 확장 대비,
   기존 라벨 유지 전제, 신규 입고분부터 적용)
3. Loyverse Integrations 애드온 활성화 → API 토큰 발급 → 매장 2곳
   연동 확인까지의 단계별 체크리스트
4. 매장 실사 재고 정산 진행 방법 (영업 중인 2개 매장 기준)
5. Loyverse 고객 4,000명 데이터 확인 항목 리스트
   (추출 가능 필드, 마케팅 활용 시 동의/규정 확인 포인트)
```
